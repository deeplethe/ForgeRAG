# OpenCraig sandbox image — Python data stack.
#
# Pinned conservatively. Each block is one capability the
# in-container agent (Hermes Agent) leans on for analysis /
# spreadsheet / PDF workflows; trimming a block means deciding
# the agent doesn't need that ability.
#
# DO NOT add ML training stacks (pytorch / transformers / xgboost).
# The default image is for analysis + visualisation, not model
# training. Heavy numerics belong in a future ``install_runtime``
# step; the base image stays small enough for fast cold start.
#
# Hermes Agent itself is installed separately via the Dockerfile
# (pinned MIT-licensed package); it brings its own loop, tools,
# and LLM client. We don't run an ipykernel inside the container —
# code execution is subprocess-based via Hermes' built-in shell
# tools, so jupyter_client / ipykernel / bash_kernel are NOT
# installed here.

# ── Agent runtime ──
# Claude Agent SDK (Anthropic, MIT) — runs in-container per turn,
# launched via ``docker exec`` by the backend's HermesContainerRunner.
# The wheel ships a self-contained ``claude`` binary in
# ``_bundled/`` (no Node.js required); the SDK's Python API wraps
# it. Built-in tools (Read / Edit / Bash / Glob / Grep / etc.)
# operate on /workdir/ which the backend bind-mounts to the user's
# private workspace. MCP-based domain tools (search / KG / library)
# are reached via HTTP back to the backend.
#
# This pin must stay aligned with the backend's pin (root
# requirements.txt) so the bundled binary's stream-json schema and
# the message-types module the backend imports stay version-locked.
claude-agent-sdk>=0.1.80
